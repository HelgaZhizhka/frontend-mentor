# Автоматическая проверка

### Как использовать скрипт для проверки

```bash
# склонировать репозиторий студента
git clone <url-репозитория> && cd <название-папки>

# скопировать скрипт в текущую директорию репозитория студента

cp /путь/к/скрипту/auto-check.sh .

# запустить скрипт автопроверки
./auto-check.sh
```

### Скрипт автоматически выполнит:

1. Проверка конфигурации:
   - tsconfig.json: strict mode
   - tsconfig.json: noImplicitAny
   - ESLint: правило no-explicit-any
   - Webpack: TypeScript loader
   - Отсутствие 'any' в коде (с локациями если найдено)

2. Статистика TypeScript:
   - Подсчёт interfaces/types (количество)
   - Подсчёт generics (количество)
   - Подсчёт enums (количество)
   - Подсчёт classes (количество)
   - Проверка access modifiers (private, public, protected)

3. Качество кода:
   - console.log (количество)
   - Закомментированный код (количество)
   - TODO/FIXME (количество)

4. Сборка и линтинг:
   - npm run lint (количество errors и warnings)
   - npm run build (проверка что собирается)

5. Git история:
   - Анализ коммитов (% Conventional Commits)
   - Проверка лишних файлов в git (node_modules, .env, dist)
   - Последние 10 коммитов

Результат: Отчёт с ✅/❌ статусами и детальной статистикой
Сохраняется в файл auto-check-report-YYYYMMDD-HHMMSS.txt

### Как использовать результаты

**Шаг 1: Проверить конфигурацию**

- Посмотреть какие проверки ✅ пройдены, какие ❌ не пройдены
- Если 'any' найден - просмотреть найденные локации в отчёте
- Проверить количество ESLint ошибок и warnings
- Оценить общее качество кода на основе статистики

**Шаг 2: Использовать статистику для ручной проверки**

- **Repository Setup:**
  - Проверить PR на GitHub (описание, ссылки, скриншот, deployment)
  - Оценить качество git commits (используя % Conventional Commits из скрипта)

- **TypeScript Features:**
  - Использовать подсчёт interfaces/generics/enums/classes как ориентир
  - Проверить **качество и осмысленность** (не только количество!)
  - Оценить архитектуру и типизацию
  - Проверить правильность использования access modifiers

- **Design & Functionality:**
  - Открыть deployment URL
  - Протестировать responsive (320px-1920px)
  - Оценить дизайн и UI/UX

**Шаг 3: Написать фидбек**

- Использовать шаблон финального комментария в чек листе
